## Core Data Structure Categories:

### 1. Intent Data Models (Move - intent.move)
- **TradeIntent<Source, Args>**: Core intent structure with `offered_resource`, `argument`, `self_delete_ref`, `expiry_time`, `witness_type`, `reservation`, and `revocable` fields
- **TradeSession<Args>**: Hot potato session type containing `argument`, `witness_type`, and `reservation`
- **FungibleAssetLimitOrder**: Trading conditions with `desired_metadata`, `desired_amount`, `issuer`, and optional `intent_id` for cross-chain linking
- **FungibleStoreManager**: Asset management with `extend_ref` and `delete_ref`

### 2. Event Structures (Move - fa_intent.move)
- **LimitOrderEvent**: Intent creation event with `intent_address`, `intent_id`, `source_metadata`, `source_amount`, `desired_metadata`, `desired_amount`, `issuer`, `expiry_time`, `revocable`
- **LimitOrderFulfillmentEvent**: Intent fulfillment event with `intent_address`, `intent_id`, `solver`, `provided_metadata`, `provided_amount`, `timestamp`
- **OracleLimitOrderEvent**: Escrow creation event with similar fields plus `min_reported_value`

### 3. Escrow Data Models (Solidity - IntentEscrow.sol)
- **Escrow struct**: Contains `maker`, `token`, `amount`, `isClaimed`, `expiry`, `reservedSolver`
- **EscrowInitialized, DepositMade, EscrowClaimed, EscrowCancelled events**

### 4. Verifier Data Models (Rust - trusted-verifier)
- **IntentEvent, EscrowEvent, FulfillmentEvent**: Normalized event structures for cross-chain monitoring
- **EventHandle, AptosEvent**: Aptos blockchain event structures
- **ApprovalSignature**: Cryptographic approval structure

### 5. Cross-Chain Data Linking
- **intent_id pattern**: How intents are linked across chains using optional `intent_id` field
- **Event correlation**: How events are matched between hub and connected chains
- **Reserved solver addressing**: How solver addresses are preserved across chains
