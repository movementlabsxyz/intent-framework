# Description: Create comprehensive use cases documentation that shows how the current system handles specific scenarios including happy path flows, error cases, edge cases, and real-world usage patterns.
# Details:
This task involves creating `docs/use-cases.md` that documents current implementation behavior across the intent framework system, organized by scenario type:

## Happy Path Use Cases:
1. **Standard Cross-Chain Swap**: Document the complete flow from intent.move (TradeIntent creation) --> escrow creation (IntentEscrow.sol) --> fulfillment (LimitOrderFulfillmentEvent) --> verifier approval --> escrow claim
2. **Oracle-Guarded Intent**: Document OracleLimitOrderEvent flow using fa_intent_with_oracle.move with OracleGuardedLimitOrder struct and signature verification
3. **Intent-as-Escrow**: Document simplified escrow flow using intent_as_escrow.move with EscrowConfig
4. **Reserved Intent**: Document solver reservation flow using intent_reservation.move with IntentToSign and IntentReserved structs

## Error Cases:
1. **Intent Expiry**: Document behavior when TradeIntent expires (EINTENT_EXPIRED) and EVM escrow expiry handling
2. **Invalid Witness**: Document EINVALID_WITNESS error from intent.move when wrong witness type provided
3. **Unauthorized Access**: Document ENOT_OWNER and ENOT_REVOCABLE errors for intent revocation attempts
4. **Cross-Chain Failures**: Document verifier signature validation failures and escrow claim rejections
5. **Token Type Mismatches**: Document ENOT_DESIRED_TOKEN errors from fa_intent.move

## Edge Cases:
1. **Non-Revocable Escrow Intents**: Document verifier requirement that escrow intents must have `revocable = false`
2. **Reserved Solver Enforcement**: Document how EVM escrow always transfers to reservedSolver regardless of claim caller
3. **Zero-Amount Cross-Chain Swaps**: Document behavior in fa_intent_cross_chain.move where source tokens aren't locked
4. **Concurrent Intent Fulfillment**: Document race conditions and prevention mechanisms

## Real-World Usage Patterns:
1. **DEX Integration**: How external systems create intents for automated market making
2. **Cross-Chain Arbitrage**: Using the framework for cross-chain price arbitrage opportunities
3. **Payment Channels**: Using intents for conditional payment releases
4. **Escrow Services**: Using intent-as-escrow for trustless P2P transactions

Each scenario should reference specific source files, struct definitions, error codes, and event emissions from the actual implementation.
